Purpose

The Student Discipline Management System (SDMS) provides a transparent, consistent and data-informed way to record, track, and resolve student discipline matters while empowering students to stay informed and participate constructively.

    Data Handling

    Purpose

    Only authorized personnel can create or update official violation records. Student access is read-focused except for appeals and messaging endpoints.

    The Student Discipline Management System (SDMS) is a centralized portal where students can monitor discipline records, read official notices, message advisers, and submit appeals.

    Roadmap (Planned)

    Core Features (Student UI)

      • Automated email / SMS notifications
      • Dashboard Cards: Total Violations, Open Cases, Last Violation, Standing.
      • Positive conduct tracking (commendations)
      • Filters: Date range + type filtering.
      • Printable clearance reports
      • Analytics: Line and bar charts (Chart.js) for trend & breakdown.
      • Parent / guardian portal
      • Violation History: Full table with status chips (Pending / Resolved).
    • Messages & Notices: Read notices; send message to adviser/prefect.
    • Appeals: Submit appeal for pending violations.

      • Responsible Use

    • Auth Guard (demo): Basic front-end check via localStorage.auth.

      • This tool supports fairness, accountability, and improvement. Use it responsibly and report any incorrect records through the appeals process.

    Version 0.1 – Student-facing module prototype

    Information Architecture (Entities)

  • Student (id, name, grade, section)
  • Violation (id, student_id, date, type, remarks, status)
  • Notice/Message (id, to_student_id, text, date, sender_role)
  • Appeal (id, student_id, violation_id, reason, submitted_at, decision, decided_at)
  • User (id, username/email, role: Student/Adviser/Discipline/Admin)

    • Front-End Data Flow

    • Login: separate page sets localStorage.auth="ok" after CAPTCHA.
    • Guard: redirects here only if authenticated.
    • Load Context: mock data now; later fetch from API.
    • Filter: in-memory filter; charts redraw with filtered set.
    • Message/Appeal: demo shows alerts; later connect to backend endpoints.

    Roles & Permissions

    • Student: View own records, message adviser, submit appeals.
    • Adviser: View class, post notices, reply to messages, recommend actions.
    • Discipline Officer: Create/update violations, set statuses, decide appeals.
    • Admin: Manage users, roles, system settings and audit logs.

    Security (when backend is added)

    • JWT sessions, hashed passwords (bcrypt/argon2).
    • RBAC on every API; audit trail for sensitive changes.
    • HTTPS-only, secure cookies/CSRF, minimal PII exposure.

    Suggested Stack & APIs

    • Front-End: HTML/CSS/JS, Chart.js
    • Back-End: Node/Express or PHP/Laravel
    • DB: MySQL/PostgreSQL
    • Endpoints: GET /violations?student_id=, GET /notices, POST /messages, POST /appeals

    Future Enhancements

    • Evidence attachments on appeals.
    • Threaded messaging, email/SMS notifications.
    • PDF/CSV exports, parent letters.
    • Admin analytics for trends and risk flags.
